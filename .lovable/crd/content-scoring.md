# CRD: 자동 발행 콘텐츠 점수 시스템

> Status: **Proposed — Future Implementation (additive, not yet shipped)**
> Owner: SearchTune OS
> Related memory: `mem://features/content-scoring`

## 0. 현재 유저 플로우와의 관계 (Compatibility Notice)

이 문서의 모든 항목은 **향후 구현 대상**이다. 현재 라이브 플로우를 변경하지 않으며, 합의 전 어떤 코드도 작성하지 않는다.

**현재 라이브 플로우 (변경 금지)**
- 단계: `draft → queued → published` (3단계, `site_posts.status` 기존값 유지)
- 발행 액션: `AutoPublish` 페이지의 "콘텐츠 발행하기" 버튼 → `publish-site-post` Edge Function
- 큐 정렬: `created_at` 내림차순
- 점수 표시: 없음 (`source_axis` 배지만 노출)
- 한도: 월 5건 (`MONTHLY_LIMIT`)

**향후 구현 항목 (이 CRD 범위 — 합의 후 별도 PR로 단계 도입)**
| 항목 | 충돌 영역 | 도입 방식 |
|---|---|---|
| 4단계 큐 (`draft→review→queued→published`) | `site_posts.status` enum 확장 | `review` 값 추가, 기존 `draft/queued/published` 의미 보존 |
| 점수 컬럼 (`draft_score`, `published_score`, `score_updated_at`) | 신규 컬럼 | nullable로 추가, 기존 SELECT/RLS 무영향 |
| `humanReview` 플래그 | 신규 컬럼 | `boolean default false`, 기존 발행 흐름 무차단 |
| `score-content` Edge Function | 신규 함수 | 단독 신설, 실패해도 발행 흐름 무영향 |
| AutoPublish 큐 배지/정렬 토글 | UI 추가 | 점수 없으면 미노출 (graceful) |
| Content 초안 점수 카드 | UI 추가 | `draft_score === null`이면 기존 UI 그대로 |
| 표준 마이크로카피 4종 | 신규 카피 | 점수 컬럼 도입 후에만 노출 |

**충돌 방지 원칙**
1. 모든 신규 컬럼은 **nullable + default**. 기존 INSERT/UPDATE 경로 변경 금지.
2. `status` enum에 `review` 추가 시 기존 코드 경로(`queued`/`published`)는 그대로 동작.
3. 점수 채점 실패는 어떤 상태 전이도 막지 않는다 (현재와 동일하게 발행 가능).
4. 표준 문구·아이콘·정렬 토글은 점수 데이터 존재 시에만 활성화 (점진적 롤아웃).
5. RLS 정책 변경 없음. 기존 `site_posts` 정책 6개 그대로 유지.
6. `publish-site-post` Edge Function 시그니처 유지 — 재채점은 백그라운드 fire-and-forget.

**도입 순서 (Sec. 9 Rollout과 동기화)**
1. 마이그레이션 (컬럼 추가) → 2. `score-content` 함수 → 3. 초안 채점 연동 → 4. UI 점수 표시 → 5. `review` status 추가 → 6. 큐 정렬/액션 매트릭스 → 7. 발행 후 재채점.

각 단계는 **이전 단계 없이도 라이브가 정상 동작**해야 한다.

## 1. Purpose (목적)
**랭킹 예측이 아닌**, 발행 전후 콘텐츠 품질 판단, 검수 우선순위 정리, 운영 재미 요소 제공.

## 2. Problem
자동/수동 생성된 글이 검색·AI 답변에 실제로 인용될 가능성을 사용자가 모른 채 발행한다.
초안 개선 유인이 약하고, 발행 큐의 우선순위를 정할 근거도 없다.

## 3. Goal
- 발행 전: 초안 점수로 개선을 유도한다.
- 발행 후: 재채점으로 최종본 품질을 추적한다.
- 큐: 낮은 점수 글을 먼저 다듬도록 정렬 옵션을 제공한다.
- 운영 재미: 품질 개선 게이미피케이션, 진척도 가시화.

## 4. Non-goals
- 검색 순위 예측/보장.
- 디자인 시스템 변경.
- 실측 조회수 기반 성과 점수 (추후 별도 과제).
- 사이트 분석 엔진의 채점 기준 변경.

## 4. Scoring Model
| 항목 | 값 |
|---|---|
| 1차 축 (User-facing) | **SEO / AEO / GEO** — 사이트 분석과 동일, 사용자 노출 라벨 |
| 2차 축 (평가 루브릭) | **검색 의도 적합성 / 제목·요약 명확성 / 본문 구조화 / 근거·사실성** — AI 채점 시 내부 루브릭 |
| 범위 | 각 0~100 정수, 총점은 1차 축 가중 평균 |
| 모델 | `google/gemini-2.5-flash` via Lovable AI Gateway |
| 결정성 | temperature = 0 |
| 입력 | title, excerpt, content(markdown), source_axis, (옵션) siteUrl |
| 출력 | `{ seo, aeo, geo, rubric, summary, tips[≤3], reasons[2~4] }` |
| 캡 규칙 | 기존 score-caps 재사용 (직접답변 부재 → AEO 캡, 출처 부재 → GEO 캡) |

### 4.1 평가 축 (2-Layer)

**1차 축 — 사용자 노출용 (UI 라벨)**
사이트 분석과 동일한 SEO/AEO/GEO 3축. 사용자에게 노출되는 점수와 배지는 항상 이 축으로 표기.

**2차 축 — AI 평가 루브릭 (내부 채점 기준)**
AI에게 더 정확한 채점을 시키기 위해 4개의 세부 루브릭으로 분해 후, 1차 축에 매핑한다.

| 루브릭 (2차 축) | 정의 | 1차 축 매핑 가중 |
|---|---|---|
| **검색 의도 적합성** (Intent Fit) | 타겟 키워드/질문에 본문이 답하는 정도, 토픽 일치도 | SEO 0.5 / AEO 0.4 |
| **제목·요약 명확성** (Title & Excerpt Clarity) | 제목 60자 내 핵심어, 요약의 직접 답변성 | SEO 0.3 / AEO 0.5 |
| **본문 구조화** (Structure) | H2 개수·계층, 문단 길이, 리스트/표 활용 | SEO 0.5 / AEO 0.2 |
| **근거·사실성** (Evidence & Factuality) | 출처 링크, 수치 근거, 인용 가능 문장 비율 | GEO 0.6 / AEO 0.2 |
| **중복·뻔한 표현 비율** (Originality) | 클리셰·상투구 비중, 동일 표현 반복도 (낮을수록 점수↑) | SEO 0.2 / AEO 0.2 / GEO 0.1 |
| **인용 친화 문장 밀도** (Quotability) | 검색엔진/AI가 이해·인용하기 좋은 자기완결형 문장 비율 | AEO 0.4 / GEO 0.4 |
| **내부 링크/주제 연결성** (Topical Connectivity) | 같은 사이트 내 관련 글 링크, 주제 클러스터 응집도 | SEO 0.3 / GEO 0.2 |
| **사람 검수 여부** (Human Review) | 사람이 검수/편집했는지 (boolean → 0 또는 100) | meta 가중 (총점 +0~5 보정) |

총점 산식 (의사코드):
```
rubric = { intentFit, titleClarity, structure, evidence,
           originality, quotability, connectivity, humanReview }   // 0~100
// 1차 축 합성 (가중 평균, weights_sum 기준 정규화)
seo = weighted_avg(intentFit:0.5, titleClarity:0.3, structure:0.5,
                   originality:0.2, connectivity:0.3)
aeo = weighted_avg(intentFit:0.4, titleClarity:0.5, structure:0.2,
                   evidence:0.2, originality:0.2, quotability:0.4)
geo = weighted_avg(evidence:0.6, originality:0.1, quotability:0.4,
                   connectivity:0.2)
base  = round(seo*0.35 + aeo*0.35 + geo*0.30)
total = clamp(base + (humanReview ? 5 : 0), 0, 100)   // 검수 보너스 +5
```
> 가중치는 초기값. 운영 데이터 기반으로 조정 (engine_config로 외부화 검토).

**왜 2-Layer인가**
- 사용자에게는 익숙한 SEO/AEO/GEO를 유지 (학습 비용 0).
- AI 채점은 더 구체적인 루브릭(8개)으로 분해해야 안정적 (deterministic + 비교 가능).
- 사유 칩(reasons)과 액션 제안(tips)은 루브릭 단위로 생성 후, 표시 시 1차 축 라벨을 붙인다.

### 4.2 제품 내 역할 (Product Roles)

점수는 단순 숫자가 아니라 운영 흐름의 **결정·정렬·추적 기준**으로 사용된다.

| 역할 | 사용처 | 설명 |
|---|---|---|
| **검수 우선순위 정렬** | 대시보드 발행 큐 / Content 목록 | 준비 점수 오름차순 정렬 = "검수 추천 순". 낮은 글이 먼저 노출되어 운영자가 우선 다듬도록 유도 |
| **발행 가능 여부 신호** | 초안 카드 라벨 | 점수대별 마이크로카피 (재작성/보강/검수 후/바로 발행) — 강제 게이팅 X, 권고만 |
| **편집 효과 측정** | 발행 후 토스트 + 대시보드 추세 | 준비 → 성장 점수 Δ 표시. 사람 검수 시 +5 보너스 자동 반영 |
| **품질 게이미피케이션** | 사이트 허브 통계 | 사이트별 평균 성장 점수, 90+ 비율 등으로 운영 동기 부여 |
| **AI 학습 피드백** | engine_update 로그 | 낮은 루브릭 통계로 차기 프롬프트 개선 힌트 추출 |

**기본 정렬 규칙 (대시보드)**
- 발행 큐 기본 정렬: `검수 추천 순 (준비 점수 오름차순)` → 낮은 점수 먼저.
- 발행 완료 목록 기본 정렬: `최신순`. 토글로 `성장 점수 높은 순` / `Δ 큰 순` 제공.
- 사람 검수 미완료(`humanReview=false`) 글은 같은 점수대에서 **상단 우선** 노출.

### 점수 2종 구분 (확정 명명)

**발행 전 — 콘텐츠 준비 점수 (Content Readiness Score)**
- 보조 라벨: **검수 추천 점수** (Review Priority Score) — 큐 정렬 컨텍스트 한정
- 시점: 초안 생성 직후
- 용도: 발행 전 개선 방향 제시, 검수 우선순위 정렬
- 저장 필드: `draft_score` (jsonb)
- 영문 변수: `readinessScore` / `reviewPriorityScore`

**발행 후 — 콘텐츠 성장 점수 (Content Growth Score)**
- 보조 라벨: **성과 건강도** (Performance Health) — 대시보드/추세 컨텍스트
- 시점: 발행 직후 재채점 (+ 추후 주기적 갱신 여지)
- 용도: 최종본 품질 추적, 초안 대비 개선 효과(Δ) 측정
- 저장 필드: `published_score` (jsonb), `score_updated_at`
- 영문 변수: `growthScore` / `performanceHealth`

> **상위 우산 명칭**은 여전히 "콘텐츠 품질 점수" (Content Quality Score). 두 점수 모두 그 하위 시점별 표현이다.

## 5. Lifecycle (4-Stage Queue)

콘텐츠는 **자동 발행이 아닌 4단계 큐**를 따른다. 사용자는 각 단계에서 점수를 보고 다음 행동을 결정한다.

| 단계 | status | 점수 시점 | 사용자 행동 | 다음 전이 |
|---|---|---|---|---|
| ① 초안 (Draft) | `draft` | 준비 점수 1차 채점 | 편집 / AI 보강 | "검수 보내기" |
| ② 검수 대기 (Review Queue) | `review` | 준비 점수 (편집 후 재채점) | 사람 검수 + `humanReview=true` | "발행 큐에 넣기" |
| ③ 발행 대기 (Publish Queue) | `queued` | 준비 점수 최종값 | 즉시 발행 / 예약 / 되돌리기 | "발행" |
| ④ 발행 완료 (Published) | `published` | 성장 점수 재채점 (비동기) | 추세 확인 / 수정 시 ②로 회귀 | — |

**전이 규칙**
- `draft → review`: 항상 가능. 권장 임계 없음 (점수 낮아도 검수 보낼 수 있음).
- `review → queued`: `humanReview=true` 필요. false면 "검수 미완료" 경고만 표시(차단 아님).
- `queued → published`: 월 5건 한도(베타) + 즉시 발행 또는 예약 시각 도래.
- `published → review`: 본문 편집 시 자동 회귀 + `published_score` 무효화 → 재발행 시 다시 성장 점수 채점.

**점수 시점**
1. **Readiness 1차** — `generate-content-draft` 응답 → `draft_score` 저장 (단계 ①).
2. **Readiness 재채점** — 본문 편집 저장 시 → `draft_score` 갱신 (단계 ②/③에서 발생).
3. **Growth** — `status: published` 전이 시 `publish-site-post`가 백그라운드 채점 → `published_score`, `score_updated_at` (단계 ④).
4. **Failure** — 채점 실패는 어떤 전이도 막지 않는다. UI는 "측정 실패" 표기만.

### 5.1 빠른 판단 기준 (Quick Decision Rules)

발행 대기/검수 대기 큐에서 **3초 안에 다음 행동을 결정**할 수 있어야 한다. 각 큐 아이템은 다음을 한 줄로 노출한다:

```
[점수 배지] [상태 라벨] · [가장 약한 축 사유] · [추천 액션]
예: 78점 · 검수 후 발행 추천 · AEO 약함(직접 답변 부재) · [검수 보내기]
```

**큐별 추천 액션 매핑**

| 단계 | 점수 구간 | 1차 추천 버튼 | 보조 버튼 |
|---|---|---|---|
| 검수 대기 | 90+ | `발행 큐로` | 편집 |
| 검수 대기 | 75~89 | `검수 완료 → 발행 큐로` | 편집 |
| 검수 대기 | 50~74 | `편집` | 발행 큐로(경고) |
| 검수 대기 | <50 | `재작성` | 삭제 |
| 발행 대기 | 90+ | `즉시 발행` | 예약 |
| 발행 대기 | 75~89 | `즉시 발행` | 검수로 되돌리기 |
| 발행 대기 | 50~74 | `검수로 되돌리기` | 강행 발행 |
| 발행 대기 | <50 | `검수로 되돌리기` | — |

**판단 보조 신호 (배지 옆 작은 아이콘)**
- 🧑 사람 검수 완료 (`humanReview=true`)
- ⚠ 캡 적용 (직접 답변 부재 / 출처 0개 등)
- ⏱ 24h 이상 큐 대기 (검수/발행 모두)
- 🔁 발행 후 재편집됨 (성장 점수 무효)

**자동 발행 정책 (명시)**
- 본 시스템은 **자동 발행을 기본값으로 하지 않는다**. `queued` 진입도 사용자 클릭 필요.
- "월 5건 한도" 등 기존 발행 제한은 ③→④ 전이에만 적용.
- 추후 옵션으로 "90+ 자동 발행 큐 진입" 토글 검토 (현재 out-of-scope).

## 6. Data (additive only)
```
site_posts:
  + draft_score        jsonb null
  + published_score    jsonb null
  + score_updated_at   timestamptz null
```
기존 컬럼/RLS 변경 없음. 마이그레이션은 별도 승인 후.

## 7. UI Surface (디자인 변경 없이)
- **Content 페이지**: draft 카드 영역에 3축 미니 점수 + 팁 1~2.
- **AutoPublish 페이지**: 큐 아이템에 점수 배지("AEO 62"), "낮은 점수 우선" 정렬 토글.
- 색상은 score-based-ui-feedback 규칙 재사용 (red <50 / orange 50~74 / blue ≥75).

### 7.1 노출 원칙: 총점만 보여주지 않는다
모든 점수 노출 지점은 다음 3계층을 함께 제공해야 한다.

1. **총점 (Headline Score)** — 0~100, 시점 라벨(준비/성장) + 색상 상태.
2. **세부 항목 (Breakdown)** — SEO·AEO·GEO 3축 점수 + 각 축의 *왜 깎였는지* 한 줄 사유.
   - 예: `AEO 62 — 도입부에 직접 답변 문장이 없습니다.`
   - 캡 적용 시 사유에 명시: `(캡 적용: 직접 답변 부재)`
3. **액션 제안 (Actionable Tips)** — 사용자가 바로 실행 가능한 제안 1~3개.
   - 형식: `[축] 행동 동사로 시작 + 구체 위치/예시`
   - 예: `[AEO] 첫 문단을 "결론부터: ..." 1~2문장으로 다시 써보세요.`
   - 예: `[GEO] 본문에 출처 링크 1개 이상 추가 (현재 0개).`
   - 예: `[SEO] H2를 5개로 줄이고 핵심 키워드를 H2에 1회 포함.`

**축출 데이터 모델 (확장)**
```ts
type AxisDetail = { score: number; reason: string; capped?: boolean };
type ActionTip   = { axis: 'seo'|'aeo'|'geo'; action: string; impact?: 'high'|'med'|'low' };
type ContentScore = {
  total: number;                 // 0~100 가중 평균
  seo: AxisDetail;
  aeo: AxisDetail;
  geo: AxisDetail;
  summary: string;               // 1~2문장 총평
  tips: ActionTip[];             // 최대 3개, impact 내림차순
};
```
이 구조는 `draft_score` / `published_score` jsonb에 그대로 저장한다.

**노출 컨텍스트별 표시량**
| 컨텍스트 | 총점 | 세부 항목 | 액션 제안 |
|---|---|---|---|
| AutoPublish 큐 배지 | ✅ | 가장 낮은 축만 1줄 | 0개 (호버/탭 시 1개) |
| Content 초안 카드 | ✅ | 3축 모두 + 사유 | 최대 2개 (high/med) |
| Content 상세/모달 | ✅ | 3축 + 캡 사유 | 최대 3개 (전체) |
| 발행 후 토스트 | ✅ + Δ | Δ가 가장 큰 축 1줄 | 0개 |

**액션 제안 작성 규칙 (AI 프롬프트)**
- 한국어, 명령형/권유형. ("~해보세요", "~를 추가하세요")
- "검색 순위 상승" 류 인과 금지. 행위 결과만 ("구조가 명확해집니다").
- 비어있을 수 있다 (100점 = `tips: []`, "검수 권장 항목 없음").

## 8. Naming & Copy (확정)
**명명 규칙 (계층)**
- **상위 우산**: "콘텐츠 품질 점수" (Content Quality Score, `contentQualityScore`) — 시스템 전체 라벨.
- **발행 전**: "콘텐츠 준비 점수" (Content Readiness Score, `readinessScore`)
  - 보조: "검수 추천 점수" (Review Priority Score, `reviewPriorityScore`) — 큐 정렬 한정
- **발행 후**: "콘텐츠 성장 점수" (Content Growth Score, `growthScore`)
  - 보조: "성과 건강도" (Performance Health, `performanceHealth`) — 대시보드/추세 한정
- "SEO 점수" 단독 표기 금지. 세 축은 항상 위 점수의 하위 지표로 표기.

**Copy**
- 발행 전 Headline: **"발행 전에 콘텐츠 준비 점수를 확인하세요."**
- 발행 후 Headline: **"발행 후 콘텐츠 성장 점수가 갱신됐어요."**
- Sub: "SEO·AEO·GEO 세 축을 합쳐 글 한 편의 인용 가능성을 점수화합니다."
- 준비 점수 배지 툴팁: "콘텐츠 준비 점수 — 발행 전 검수 우선순위 (0~100)"
- 성장 점수 배지 툴팁: "콘텐츠 성장 점수 — 발행본 구조 품질 추세 (0~100)"
- 정렬 토글: "검수 추천 순 (낮은 점수 우선)"
- 낮은 준비 점수: "지금 5분만 다듬으면 준비 점수를 올릴 수 있어요."
- 재채점 결과: "성장 점수가 갱신됐어요. 준비 점수 대비 +N점."

**점수대별 마이크로카피 — 표준 문구 (canonical, 변경 금지)**

형식: `{점수}점 · {라벨}` (가운뎃점 ` · ` 구분, 한국어, 점수 앞뒤 공백 없음)

| 점수 구간 | 표준 라벨 | 예시 | 색상 |
|---|---|---|---|
| 90~100 | `바로 발행 가능` | **92점 · 바로 발행 가능** | blue |
| 75~89  | `검수 후 발행 추천` | **78점 · 검수 후 발행 추천** | blue |
| 50~74  | `근거 보강 필요` | **63점 · 근거 보강 필요** | orange |
| 0~49   | `재작성 권장` | **41점 · 재작성 권장** | red |

> 이전 문구(`검수 불필요 · 바로 발행 OK`, `보강 후 발행 권장`)는 폐기. 위 4개 라벨이 유일한 표준.

**약점 축 기반 변형 (50~74 구간 한정, 선택적 사용)**
가장 낮은 축이 명확할 때 `근거 보강 필요`를 더 구체적인 변형으로 교체 가능:
- GEO 최저 → `근거 보강 필요` (기본, 출처/수치 부재)
- AEO 최저 → `직접 답변 보강 필요`
- SEO 최저 → `구조 보강 필요`

**컨텍스트별 적용**
- 큐 배지 기본 표기: `{점수}점 · {라벨}` 한 줄.
- 카드/모달에서는 라벨 아래 가장 약한 축 사유를 한 줄 더 노출 (예: "AEO가 가장 낮아요 — 직접 답변 부재").
- 발행 후 컨텍스트에서는 Δ를 점수 옆에 붙임: `78점 (+6) · 검수 후 발행 추천`.
- "추천/권장/필요"는 강제가 아님. 사용자는 어느 점수든 발행 가능.

**점수 사유 칩 (Reason Chips) — 카드/모달에서 라벨 아래 노출**
점수 옆에는 "왜 이 점수인가"를 보여주는 짧은 사유 칩 2~4개를 함께 표기한다.
긍정 사유(✓)와 보강 사유(△)를 섞어 균형감 있게 노출.

예시 (78점 · 검수 후 발행 추천):
- ✓ 구조 좋음
- ✓ 제목 명확
- △ 근거 문장 보강 필요
- △ 사람 검수 미완료

**칩 데이터 모델 (확장)**
```ts
type ReasonChip = {
  kind: 'positive' | 'improve' | 'blocker';  // ✓ / △ / ⚠
  label: string;                              // 짧은 한국어, 6~12자 권장
  axis?: 'seo'|'aeo'|'geo'|'meta';           // meta = 검수 상태 등 비-축
};
type ContentScore = {
  total, seo, aeo, geo, summary,
  tips: ActionTip[],
  reasons: ReasonChip[];   // 2~4개, positive ≥1, improve ≥1 권장
};
```

**칩 작성 규칙**
- 명사구 위주, 6~12자. 동사 금지(동사는 액션 제안에서 다룬다).
- positive: `구조 좋음`, `제목 명확`, `JSON-LD 포함`, `직접 답변 도입부`
- improve: `근거 문장 보강 필요`, `H2 과다`, `출처 링크 부족`, `예시 부족`
- blocker(드물게): `직접 답변 부재 (AEO 캡)`, `출처 0개 (GEO 캡)`
- meta 축 예: `사람 검수 미완료`, `초안 상태`, `수정 후 미재채점`
- 노출 우선순위: blocker → improve(impact 높은 순) → positive

**컨텍스트별 칩 노출량**
| 컨텍스트 | 칩 개수 |
|---|---|
| 큐 배지 | 0 (호버 시 1) |
| 초안 카드 | 2~3 |
| 상세/모달 | 2~4 (전체) |
| 발행 후 토스트 | 1 (Δ 큰 축의 사유) |

**오해 방지 (Anti-misleading) — 필수**
- 금지: "검색 1위", "상위 노출 보장", "순위 상승 점수", "랭킹 점수", "구글 점수", "SEO 보장".
- 권장: "인용 가능성", "검수 우선순위", "콘텐츠 구조 품질", "AI 답변 채택 준비도".
- 모든 점수 노출 옆/툴팁에 고지 1줄 동반: "※ 검색 순위를 보장하지 않습니다. 콘텐츠 구조 품질 지표예요."
- 도움말/모달 긴 형: "이 점수는 글의 구조·답변 명확성·인용 친화도를 평가한 내부 지표입니다. 실제 검색 순위나 트래픽을 보장하지 않으며, 검색 엔진 알고리즘과 무관하게 산정됩니다."
- 100점 = "완벽" 아님. "검수 권장 항목 없음"으로 설명.
- 점수 변화 메시지에 "순위 상승" 류 인과 표현 금지. 행위 결과만 기술("구조가 개선되었어요").
- 최초 1회 info 아이콘으로 긴 형 고지 노출 + dismiss 가능.

### 11.1 상태값 (status enum) 초안
`site_posts.status` 확장값. 기존 값(`draft`, `queued`, `published`)의 의미는 **그대로 유지**하고 `review`만 신설.

```ts
type SitePostStatus =
  | 'draft'      // ① 초안 — 작성 중, 채점 1차 (기존)
  | 'review'     // ② 검수 대기 — 사람 확인 필요 (신규)
  | 'queued'     // ③ 발행 대기 — 검수 완료, 발행 예약/대기 (기존, 의미 좁힘)
  | 'published'  // ④ 발행 완료 — 라이브, 성장 점수 추적 (기존)
  | 'archived';  // 보존용 (기존, 변경 없음)
```

**한국어 매핑 (UI 라벨)**
| status | 라벨 (배지) | 짧은 설명 (툴팁/모달) |
|---|---|---|
| `draft` | `초안` | "작성 중인 글이에요. 검수 보내기 전입니다." |
| `review` | `검수 대기` | "사람이 한 번 더 확인할 차례예요." |
| `queued` | `발행 대기` | "검수가 끝났고, 발행만 누르면 라이브됩니다." |
| `published` | `발행 완료` | "라이브 중이에요. 성장 점수로 추세를 봅니다." |
| `archived` | `보관` | "목록에서 숨긴 글입니다." |

### 11.3 큐 화면 카피 초안
**섹션 헤딩**
- `검수 대기 ({count})`
- `발행 대기 ({count})`
- `발행 완료 ({count})`

**빈 상태 (Empty state)**
- 검수 대기 0건: `검수할 초안이 없어요. 새 글을 작성하거나 추천을 받아보세요.`
- 발행 대기 0건: `발행 대기 중인 글이 없습니다. 검수 완료된 글이 여기로 와요.`
- 발행 완료 0건: `아직 발행된 글이 없습니다.`

**액션 버튼 라벨 (단계별)**
| 단계 | 1차 버튼 | 보조 버튼 |
|---|---|---|
| 초안 (`draft`) | `검수로 보내기` | `편집` / `삭제` |
| 검수 대기 (`review`) | `검수 완료 → 발행 큐로` | `편집` / `초안으로` |
| 발행 대기 (`queued`) | `즉시 발행` | `예약 발행` / `검수로 되돌리기` |
| 발행 완료 (`published`) | `보기` | `편집` (편집 시 자동 `review` 회귀) |

**경고/확인 다이얼로그 카피**
- 점수 <50 발행 강행: `점수가 낮아요 ({score}점 · 재작성 권장). 그래도 발행할까요?`
- `humanReview=false`로 발행 큐 진입: `사람 검수가 완료되지 않았어요. 그래도 발행 큐로 보낼까요?`
- `published → review` 회귀: `발행본을 편집하면 검수 대기로 돌아가고, 성장 점수가 다시 측정됩니다.`
- 월 한도 도달: `이번 달 발행 한도(5건)를 모두 사용했어요. 다음 달에 다시 시도해 주세요.`

**상태 변경 토스트**
- `→ review`: `검수 대기로 보냈어요.`
- `→ queued`: `발행 큐에 추가했어요.`
- `→ published`: `발행되었어요. 성장 점수는 잠시 후 갱신됩니다.`
- 성장 점수 갱신: `성장 점수가 갱신됐어요. 준비 점수 대비 {±N}점.`

### 11.4 점수 설명 문구 초안 (도움말 / info 아이콘 / 모달)

**짧은 형 (툴팁, 1줄)**
- 콘텐츠 준비 점수: `발행 전 검수 우선순위를 알려주는 0~100 지표예요.`
- 콘텐츠 성장 점수: `발행본의 구조 품질을 추적하는 0~100 지표예요.`
- 검수 추천 점수(=준비 점수, 큐 정렬 맥락): `점수가 낮을수록 먼저 다듬으면 좋아요.`
- 고지: `※ 검색 순위를 보장하지 않습니다. 콘텐츠 구조 품질 지표예요.`

**중간 형 (카드 하단 부연, 2~3줄)**
> SEO·AEO·GEO 세 축을 합쳐 글 한 편의 인용 가능성을 점수화해요.
> 점수가 낮을수록 먼저 다듬을 우선순위가 높습니다.
> 검색 순위나 트래픽을 보장하지 않으며, 검색 알고리즘과는 무관합니다.

**긴 형 (도움말 모달, 최초 1회 노출)**
> **콘텐츠 품질 점수란?**
>
> 글 한 편이 검색엔진과 AI 답변 엔진에 **얼마나 잘 인용될 수 있는 구조인지**를 0~100으로 평가한 내부 지표입니다. 세 가지 축으로 나뉩니다.
>
> - **SEO** — 검색엔진이 본문을 이해하고 색인할 준비도
> - **AEO** — AI 답변 엔진이 직접 답변으로 채택할 준비도
> - **GEO** — 생성형 AI(ChatGPT, Perplexity 등)가 출처로 인용할 준비도
>
> 점수는 두 시점에 측정됩니다.
> - **콘텐츠 준비 점수** (발행 전): 초안에서 개선 방향과 검수 우선순위를 알려줍니다.
> - **콘텐츠 성장 점수** (발행 후): 발행본의 구조 품질을 추적합니다.
>
> 이 점수는 **검색 순위나 방문자 수를 보장하지 않습니다.** 검색엔진 알고리즘과 무관하게 글의 구조·답변 명확성·인용 친화도만 평가합니다. 100점은 "완벽"이 아니라 "검수 권장 항목 없음"을 뜻합니다.

### 11.5 사유 칩 (Reason Chips) 사전 — 권장 라벨 풀
AI 채점 응답이 자유 텍스트로 칩을 만들 때 일관성을 유지할 수 있도록 표준 라벨 풀을 둔다. AI는 가능한 한 이 풀에서 선택하고, 새 표현이 필요할 때만 자유 생성한다.

**positive (✓, 6~12자 명사구)**
`구조 좋음` · `제목 명확` · `요약 명확` · `직접 답변 도입부` · `JSON-LD 포함` · `출처 충분` · `예시 풍부` · `H2 위계 정돈` · `인용 친화 문장 다수` · `내부 링크 충분`

**improve (△)**
`근거 문장 보강 필요` · `직접 답변 부재` · `H2 과다` · `H2 부족` · `출처 링크 부족` · `예시 부족` · `요약 모호` · `중복 표현 다수` · `내부 링크 부족` · `문단 과장`

**blocker (⚠, 캡 적용 시만)**
`직접 답변 부재 (AEO 캡)` · `출처 0개 (GEO 캡)` · `본문 과소 (전체 캡)`

**meta (검수/상태 관련)**
`사람 검수 미완료` · `초안 상태` · `수정 후 미재채점` · `발행 후 미편집`

### 11.6 액션 제안 (Tips) 템플릿 초안
`[축] 동사로 시작 + 구체 위치/예시` 형식. AI가 채울 placeholder는 `{}`로 표기.

- `[AEO] 첫 문단을 "결론부터: {핵심답변}" 1~2문장으로 다시 써보세요.`
- `[AEO] 핵심 질문 "{질문}"에 대한 직접 답변을 도입부에 배치하세요.`
- `[GEO] 본문에 출처 링크 {N}개 이상 추가 (현재 {current}개).`
- `[GEO] 수치 근거를 인용형 문장으로 1~2개 보강하세요. (예: "{예시문}")`
- `[SEO] H2를 {N}개로 정돈하고 핵심 키워드 "{kw}"를 H2에 1회 포함하세요.`
- `[SEO] 제목을 60자 이내로 줄이고 핵심어 "{kw}"를 앞쪽에 배치하세요.`
